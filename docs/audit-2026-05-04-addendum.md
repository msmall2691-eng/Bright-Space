## API contract corrections

### `POST /api/jobs/admin/rehydrate-job-dates-from-gcal`

**The brief implied this endpoint accepts `dry_run` and `limit` parameters. It does not.**

What actually happens:

- The endpoint silently ignores any JSON body. Sending `{"dry_run": true, "limit": 10}` is identical to sending `{}` — both cause a real run over every job with a `gcal_event_id`.
- The response includes `"dry_run": false` regardless of what you pass in.
- It processes *all* jobs with a `gcal_event_id` in one call. There is no batching.

**Verified response shape (live, May 4 2026):**

```json
{
  "dry_run": false,
  "errors": [],
  "sample_updates": [
    { "job_id": 52, "scheduled_date": "2026-04-23", "start_time": "09:00:00",
      "end_time": "12:00:00", "source": "gcal_instance" }
  ],
  "skipped_already_populated": 0,
  "skipped_no_gcal_id": 0,
  "total_jobs_checked": 63,
  "updated": 63
}
```

**Implication for Claude Code:** if you want a real dry-run capability, that's a code change to add — don't rely on the existing flag. For now, treat every call as state-mutating.

### `POST /api/visits/admin/backfill-visits-from-jobs`

**The brief implied this endpoint accepts a body. It does not.**

- No parameters. No query string, no JSON body, no flags. Just `POST` with auth.
- Idempotent: skips any job that already has a `Visit` linked.
- No side effects: does not push to GCal, does not send SMS, does not send email.
- Per-job try/except (visits_router.py:227) — failures collect into `errors[]` without aborting the rest.
- ~6 seconds wall-clock for 69 inserts on the prod DB on May 4 2026.

**Verified response shape (live, May 4 2026):**

```json
{
  "created": 69,
  "errors": [],
  "skipped": 1,
  "total_jobs": 70,
  "message": "Backfill complete: created 69 visits, skipped 1 existing"
}
```

### `GET /api/visits/admin/coverage-check`

Behaves as documented. Read-only. Returns:

```json
{
  "coverage_percent": 100,
  "healthy": true,
  "jobs_without_visits": 0,
  "total_jobs": 70,
  "total_visits": 70
}
```

The `healthy` flag is what the Schedule banner gates on. When `healthy: false`, the boot self-diagnostic also prints `VISITS COVERAGE DRIFT: ...` to stdout on every container start — useful for monitoring.
