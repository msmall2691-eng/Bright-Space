# BrightBase — Process, UX, and Schema Audit

**Audit date:** Apr 23, 2026
**Target:** https://brightbase-production.up.railway.app
**Company:** The Maine Cleaning Co. (residential + STR turnover cleaning)
**Intended reader:** Claude Code (implementation partner)

This document is an end-to-end audit of BrightBase's current workflow, UX/UI, data model, and API surface, with a before/after process map and a concrete implementation plan. It is written to be executed as a series of PRs by Claude Code.

## 4. Future-state target (modern cleaning-company SaaS baseline)

Think ServiceTitan / Jobber / Housecall Pro, but native to STR-heavy operators. The mental model that matches real cleaning-company operations:

```
 Lead ─▶ Opportunity ─▶ Quote ─▶ (Accepted) ─▶ Job(s) ─▶ Visit(s) ─▶ Invoice ─▶ Payment
                                             │
                                             ├─ attached to Property (1:N)
                                             ├─ optional RecurringSchedule (1:N)
                                             └─ optional iCal Source (STR)

 Cleaner / Crew  ◀── Dispatch ──▶ Visit  (one job can span multiple visits)
```

### Key shifts vs. today

1. **Job vs. Visit split.** One job = "do the cleaning at X property until Y date"; each physical attendance is a `Visit` with its own date, crew, checklist, and photos. Recurring cleanings become one job with many visits. STR turnovers become one job per booking with (usually) one visit. This is the #1 structural change.
2. **Property as the hub, not the address string.** Every job is anchored to a Property. One-off residential cleans auto-create a Property on first visit.
3. **Canonical address on Property only.** Clients/quotes/jobs reference properties; no duplicated address strings.
4. **Proper types.** Dates are `DATE`, times are `TIME`, timestamps are `TIMESTAMPTZ`. No more VARCHAR dates.
5. **State machines, not booleans.** Jobs and visits have explicit statuses with transitions; side effects (GCal, SMS, invoice) are logged in an audit table with timestamps + error text.

## 5. Proposed schema changes

Target Postgres. Additive first so Claude Code can ship in small PRs without breaking reads.

### 5.1 Fix data types (PR 1 — high priority)

```sql
-- jobs.scheduled_date currently VARCHAR. Must become DATE.
ALTER TABLE jobs
  ADD COLUMN scheduled_date_new DATE;
UPDATE jobs
  SET scheduled_date_new = CASE
    WHEN scheduled_date ~ '^\d{4}-\d{2}-\d{2}$' THEN scheduled_date::date
    ELSE NULL
  END;
-- verify: SELECT COUNT(*) FROM jobs WHERE scheduled_date IS NOT NULL AND scheduled_date_new IS NULL;
ALTER TABLE jobs DROP COLUMN scheduled_date;
ALTER TABLE jobs RENAME COLUMN scheduled_date_new TO scheduled_date;
CREATE INDEX idx_jobs_scheduled_date ON jobs(scheduled_date);

-- start_time / end_time: convert VARCHAR -> TIME
-- accepted_at / created_at / updated_at: should be TIMESTAMPTZ
```

After this lands, remove the Python date-arithmetic shim added in deploy `7ca89b2f` and use native `scheduled_date - INTERVAL '1 day'` again.

### 5.2 Introduce `Visit` (PR 2 — medium)

```sql
CREATE TABLE visits (
  id BIGSERIAL PRIMARY KEY,
  job_id BIGINT NOT NULL REFERENCES jobs(id) ON DELETE CASCADE,
  sequence INT NOT NULL,          -- 1st visit, 2nd visit, ...
  scheduled_date DATE NOT NULL,
  start_time TIME NOT NULL,
  end_time TIME NOT NULL,
  status TEXT NOT NULL DEFAULT 'scheduled',   -- scheduled|dispatched|en_route|in_progress|completed|canceled|no_show
  cleaner_ids BIGINT[] DEFAULT '{}',
  gcal_event_id TEXT,
  ical_source TEXT,                -- which iCal feed this originated from (if STR)
  ical_uid TEXT,                   -- RFC 5545 UID for idempotency
  checklist_template_id BIGINT,
  completed_at TIMESTAMPTZ,
  completed_by BIGINT REFERENCES users(id),
  notes TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (job_id, sequence),
  UNIQUE (ical_source, ical_uid)   -- iCal idempotency — never duplicate a booking
);
CREATE INDEX idx_visits_date ON visits(scheduled_date);
CREATE INDEX idx_visits_status ON visits(status);
```

`jobs` becomes the long-lived container (service agreement); `visits` are the physical events on the calendar. Start by backfilling one visit per job so the UI keeps working.

### 5.3 Tighten state on existing tables (PR 3)

```sql
-- jobs.status
ALTER TABLE jobs ADD CONSTRAINT jobs_status_chk
  CHECK (status IN ('draft','active','paused','completed','canceled'));

-- quotes.status explicit state
ALTER TABLE quotes ADD COLUMN viewed_at TIMESTAMPTZ;
ALTER TABLE quotes ADD COLUMN declined_at TIMESTAMPTZ;
ALTER TABLE quotes ADD COLUMN follow_up_sent_at TIMESTAMPTZ;
ALTER TABLE quotes ADD CONSTRAINT quotes_status_chk
  CHECK (status IN ('draft','sent','viewed','accepted','declined','expired','converted'));

-- intake outcomes
ALTER TABLE intakes ADD COLUMN converted_quote_id BIGINT REFERENCES quotes(id);
ALTER TABLE intakes ADD COLUMN converted_client_id BIGINT REFERENCES clients(id);
ALTER TABLE intakes ADD COLUMN archived_at TIMESTAMPTZ;
ALTER TABLE intakes ADD COLUMN first_response_at TIMESTAMPTZ;  -- SLA tracking
```

### 5.4 Canonicalize address via Property (PR 4)

- Make `jobs.property_id` **required** via a two-step migration:
  1. Auto-create a `Property` for each client with at least one job that has a free-text address. Name = address first line; `property_type` inferred from `job.job_type`.
  2. Backfill `jobs.property_id` for all 26 jobs currently missing it.
  3. Then `ALTER TABLE jobs ALTER COLUMN property_id SET NOT NULL`.
- Drop the denormalized `jobs.address` column once UI reads from `properties.address`.

### 5.5 Integration audit log (PR 5)

```sql
CREATE TABLE integration_events (
  id BIGSERIAL PRIMARY KEY,
  entity_type TEXT NOT NULL,       -- 'job','visit','quote','invoice'
  entity_id BIGINT NOT NULL,
  provider TEXT NOT NULL,          -- 'gcal','gmail','sms','connecteam','stripe'
  action TEXT NOT NULL,            -- 'create','update','delete','send','sync'
  status TEXT NOT NULL,            -- 'ok','failed','retrying'
  external_id TEXT,                -- gcal_event_id, message-id, etc.
  error TEXT,
  request JSONB,
  response JSONB,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_integration_entity ON integration_events(entity_type, entity_id);
```

Replace `calendar_invite_sent bool` and `sms_reminder_sent bool` with queries against this table. Makes retries, observability, and SLA reporting trivial.

### 5.6 Bonus: explicit iCal source table

```sql
CREATE TABLE property_ical_feeds (
  id BIGSERIAL PRIMARY KEY,
  property_id BIGINT NOT NULL REFERENCES properties(id) ON DELETE CASCADE,
  url TEXT NOT NULL,
  label TEXT,                      -- 'airbnb','vrbo','hospitable','manual'
  timezone TEXT NOT NULL DEFAULT 'America/New_York',
  last_synced_at TIMESTAMPTZ,
  last_sync_status TEXT,
  last_sync_error TEXT,
  active BOOLEAN NOT NULL DEFAULT true,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

Today there's a singular `properties.ical_url` + an `icals` array-ish field. Splitting this out makes multi-feed STR properties (AirBNB + VRBO on the same house) work cleanly, gives per-feed sync status, and the `last_sync_error` field ends the "I don't know why it didn't import" class of support tickets.

## 6. API changes

| Endpoint | Change | Why |
|---|---|---|
| `POST /api/quotes/{id}/convert-to-job` | Write `job.quote_id = {id}`; update `quote.status = 'converted'`; return the created `{job_id, visit_ids[]}`. | Fixes the missing back-link. |
| `POST /api/quotes/public/{token}/accept` | Set `quote.accepted_at = now()`, `quote.status = 'accepted'`; trigger a domain event `QuoteAccepted` → auto convert-to-job **only if** the quote is already tied to a Property. Otherwise route to a "needs scheduling" queue. | One less admin click. |
| `GET /api/quotes/public/{token}` | Also record `viewed_at` on first GET (IP/UA-stamped). | Enables "client saw it 3 days ago and hasn't clicked" follow-ups. |
| `POST /api/intake/{id}/convert-to-quote` | Also create / link `client_id`, `property_id`, and stamp `intakes.converted_quote_id`. | Traceability. |
| `GET /api/jobs` | Add `?status`, `?date_from`, `?date_to`, `?property_id`, `?cleaner_id`, `?job_type` filters; return a `visits` summary. | The schedule and dashboards need server-side filtering, not all 57. |
| `POST /api/visits` / `PATCH /api/visits/{id}` | New. Mutations on the visit, not the job. | Matches the split in §5.2. |
| `POST /api/visits/{id}/start` / `.../complete` | New. With optional photo upload + checklist result + signature. | Lifecycle tracking. |
| `POST /api/properties/{id}/ical-feeds` | New. Manage feeds per §5.6. | Multi-feed STR. |
| `GET /api/integrations/events` | New. Filterable audit view. | Observability + support. |
| `POST /api/dispatch/preview` | New. Given a date range, return "who goes where" with conflict detection. | Removes dispatch guesswork. |

Security: all routes that currently require auth should also check `role` — a cleaner should not be able to `GET /api/clients`. A simple per-endpoint role policy doc should be produced as part of PR 3.

## 7. UI / UX redesign

### 7.1 Information architecture

Current sidebar: **Dashboard · Workspace · Requests · Quoting · Invoicing · Clients · Schedule · Properties · Comms**.

Proposed:

- **Today** (replaces Dashboard for day-of operations; role-aware)
- **Pipeline**
  - Requests → Quotes → Jobs (unified funnel view with stage filters — combines the 3 sales-pipeline items today)
- **Schedule**
- **Properties** (first-class, because STR is half the business)
- **Clients**
- **Invoices**
- **Inbox** (renames Comms; merges SMS + Gmail threads + internal notes into one timeline per client)
- **Workspace / AI** (keep the existing Ask AI)
- **Admin** (settings, integrations, team, automations)

### 7.2 High-impact UX changes

1. **Visit-centric Schedule.** Each block on the calendar is a Visit with a colored left border by property type (residential / STR / commercial), a small avatar for the assigned cleaner, and an icon column (📅 GCal synced · 📲 reminder sent · ✅ completed · 📸 photos uploaded). Clicking opens a side drawer (not a modal) with the full visit record, client, property link, checklist, photos, dispatch status, and GCal event link.
2. **"New Job" → guided flow.** Replace the single modal with a 3-step wizard: *Who* (client + property, with an inline "add new" fork) → *What* (template: one-off residential / recurring residential / STR turnover / commercial) → *When* (date range, recurrence pattern, first visit time). Each step is a URL slug so it's deep-linkable and cancellable without data loss.
3. **Property drawer with booking feed.** On a Property, show the most recent iCal pulls from each feed with status chips (✅ synced 2 min ago / ⚠️ failed 3x / ⏸ paused). One click per feed to "sync now" and a toggle to pause. Show a combined occupancy ribbon so the dispatcher sees guest check-in/check-out alongside the turnover.
4. **Quote reader for admins.** Replace the raw Edit Quote form with a two-pane view: the editable spec on the left, a live WYSIWYG of the client-facing page on the right. Today's public page is already good — the admin just can't see it while drafting.
5. **One-tap actions from Today.** Each card has a primary CTA (Start, Complete, Reschedule, Message Client, Call Client). Mobile-first sizing — cleaners use this on their phones.
6. **State chips everywhere, booleans nowhere.** Replace yes/no columns with timestamped chips ("invite sent 2h ago", "no reminder — not due yet"). Hover reveals the `integration_events` row behind it.
7. **Unified client timeline.** On a client, merge SMS, email threads, quotes sent, jobs scheduled, and internal notes into one reverse-chronological timeline. Today this is scattered across Requests + Comms + Quoting.
8. **Empty states with actions.** The Invoicing KPI shows `$0` today. Empty states should offer a "Generate invoice from a completed visit" CTA with counts of eligible visits.
9. **Keyboard-first.** `Cmd+K` command palette for "schedule a visit", "open client", "find unsent quote". The app already has "Ask AI ⌘K" — extend it.
10. **Accessibility pass.** Ensure all primary CTAs have explicit labels (some icons were ambiguous), focus rings are visible, color alone isn't the only state indicator.

### 7.3 Visual direction

Current look is clean, modest, Tailwind-ish. To match a modern cleaning-ops SaaS:

- Keep the blue/green base; introduce a **service-type palette** (residential = blue, STR = amber, commercial = purple) consistently on badges, calendar blocks, and property cards.
- Density modes: "Comfortable" (office desktop) and "Compact" (dispatcher with 2 monitors).
- Use color-weighted status chips, not colored backgrounds on rows — easier to scan.
- Card layouts that follow the 8-point grid; switch to CSS grid for the calendar to enable overlapping visits.
- Motion: subtle 150ms transitions on drawer open, list reorder, state-chip flips. Nothing else moves.

## 8. Before / after — the five key journeys

### Journey A: New inbound SMS → scheduled cleaning

| Stage | Before (today) | After | Time saved |
|---|---|---|---|
| SMS arrives | Appears in Requests, `status=new` | Same | — |
| Triage | Admin opens request manually | Auto-classified (residential / STR / commercial); AI-drafted reply; 1-click "Accept classification" | 3–5 min |
| Client record | Admin clicks "View client" | Client row auto-created or deduped on first SMS; Property auto-created on first valid address | 2 min |
| Quote draft | Admin clicks Create Quote → form | AI pre-fills line items from matching past quotes / templates; admin reviews | 5–10 min |
| Send | Admin clicks Send | Same | — |
| Accept | Client clicks Accept; admin later clicks Convert | Accept auto-converts to Job + Visit if property is set; GCal + SMS reminder queued | 4 min |
| Schedule change | Admin edits job modal | Drag the Visit on the schedule; everything else reacts | 2 min |
| **Total** | ≈ 16–23 min | ≈ 3–5 min | **~75% reduction** |

### Journey B: New STR booking hits iCal

| Stage | Before | After |
|---|---|---|
| iCal sync | Cron job, fails silently if format varies | Feed-level job with visible sync status + last error |
| Date parsing | Relied on SQL INTERVAL arithmetic, broke on VARCHAR | DATE-typed `scheduled_date` + clean Python parsing |
| Dedupe | None explicit | `UNIQUE(ical_source, ical_uid)` on Visit |
| Dispatch | Manual | Default cleaner per property; alerts on conflict |

### Journey C: Visit day

| Stage | Before | After |
|---|---|---|
| Cleaner views work | Separate Connecteam | Mobile-friendly `/today` in BrightBase with day's visits + directions + house codes |
| Start job | No signal | Tap "Start"; stamps `visits.started_at`, sends "cleaner is on-site" webhook |
| Complete | Admin later marks manual | Tap "Complete"; uploads checklist + before/after photos; invoice auto-drafted |

### Journey D: Recurring residential

| Stage | Before | After |
|---|---|---|
| Setup | Single recurring_schedule row; jobs duplicated individually | One Job with a recurrence rule (RRULE); Visits auto-generated 8 weeks out |
| Skip / reschedule | Edit each job | Skip this visit only / Push forward / End series — all first-class |
| End date | Not modeled | `job.ends_at` explicit |

### Journey E: Follow-up on unread quote

| Stage | Before | After |
|---|---|---|
| Did the client open it? | Unknown | `quote.viewed_at` timestamp |
| Nudge | Admin remembers to text | Rule: if `sent 48h ago and not viewed` → auto-SMS template; if `viewed 24h and not accepted` → different template |

## 9. Implementation plan (PR roadmap)

1. **PR 1 — Data types (1 day).** Migrate `jobs.scheduled_date` VARCHAR → DATE + indexes; do the same for `start_time`/`end_time`; normalize timestamp columns to TIMESTAMPTZ. Remove the Python date-arithmetic shim.
2. **PR 2 — Property linkage (1 day).** Backfill properties from unlinked jobs; make `jobs.property_id` NOT NULL; drop `jobs.address`.
3. **PR 3 — Quote link + state (0.5 day).** Convert-to-job writes `job.quote_id`; public accept sets `accepted_at`; add `viewed_at`.
4. **PR 4 — Visits table + backfill (2 days).** Ship behind a feature flag; UI still reads Jobs by default.
5. **PR 5 — Integration events log (1 day).** New table + adapter in gcal/sms/gmail paths; replace booleans in reads.
6. **PR 6 — iCal feeds table + sync status UI (1 day).**
7. **PR 7 — UI: side-drawer replaces Edit Job modal; adds property, cleaner, dispatch, gcal chips (2 days).**
8. **PR 8 — UI: schedule → Visit-centric (2 days).**
9. **PR 9 — UI: 3-step New Job wizard (1 day).**
10. **PR 10 — UI: unified client timeline (2 days).**
11. **PR 11 — Today dashboard for cleaners (mobile) (2 days).**
12. **PR 12 — Quote follow-up automation rules (1 day).**

**Total**: ≈ 17 engineer-days. Can ship to users after PR 3; PRs 4–12 are additive.

## 11. Risk & rollback

- **Type migrations** are the highest-risk change. Run on a Railway preview DB first, diff row counts, keep the old column name available as a generated column for one release cycle.
- **Making `property_id` NOT NULL** must happen after backfill. Do it in a separate migration with a single `ALTER TABLE` transaction.
- **Visits table** behind a feature flag. The old `/api/jobs` response keeps shape; `visits[]` is added alongside. UI opts in per-view.
- **Quote auto-accept → convert** needs an idempotency key to prevent double conversion (same accept POST hitting twice from the client).
