You already have `jobs.gcal_event_id` and a `rehydrate-job-dates-from-gcal` endpoint. Lean into it.

## P2 â€” Code-cleanup quick wins from the audit (parallel work, ~half day)

These come straight out of the repo audit.

1. **Decouple Job/Visit dual-write.** `backend/modules/scheduling/router.py` creates a Job and then writes a primary Visit in the same handler. Extract to a single `create_job_with_primary_visit(...)` service function with an idempotency key, so partial failures stop orphaning jobs (root cause of the 69-orphan problem).
2. **Paginate `_sync_ical_url`.** `backend/integrations/ical_sync.py` parses and commits an entire feed in one transaction. For 100+ events this will lock and timeout. Batch in groups of 25.
3. **Convert `source` / `status` string fields to enums.** `PropertyIcal.source`, `Visit.ical_source`, `Job.status` are all strings. Define real Postgres ENUMs (or Python `Enum`s plus CHECK constraints) so we get typed dropdowns on the frontend automatically.
4. **Index `(jobs.id, visits.job_id)` and `visits.scheduled_date`.** Coverage-check and schedule queries scan today.
5. **DRY the `visit_to_dict` / `job_to_dict` enrichers** in `visits_router.py` and `scheduling/router.py` into shared serializers.
